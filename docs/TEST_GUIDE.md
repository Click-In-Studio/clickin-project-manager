# 测试指南

本文档面向 QA / 人工测试人员。

**自动化单元测试**（API 鉴权、数据库完整性、权限逻辑）由 CI 保证，**不是本文档的覆盖范围**。人工测试的目标是：**UI 交互正确性**和**前端逻辑正确性**——这是自动化测试无法替代的部分。

## 1. Sanity Check — 了解自动化测试已覆盖什么

在开始人工测试之前，先确认 **CI 已通过**，然后明确自动化已覆盖的内容，从而知道人工测试**不需要**重复验证什么。

### 1.1 CI 自动覆盖的内容

打开对应 PR 的 GitHub Actions，确认以下两个 job 全部绿色：

| CI Job | 覆盖内容 |
|--------|---------|
| `lint-and-typecheck` | TypeScript 类型安全、ESLint 规范 |
| `unit-test` | API 鉴权（401/403）、输入验证（400）、DB 完整性、权限逻辑、Migration 正确性 |

**如果 CI 是红的，停止人工测试，等开发修好再继续。**

### 1.2 自动化测试已覆盖，人工无需重复验证的内容

- 未登录访问 API → 自动返回 401（单元测试已覆盖）
- 非成员访问他人剧目 API → 自动返回 403
- 空字符串、缺少必填字段 → 自动返回 400
- 数据库级别的重复 key、孤儿行、级联删除
- Migration 前后数据映射的 FK 一致性

### 1.3 人工测试的明确边界

自动化测试不覆盖：

- **UI 渲染**：组件是否正确显示，数据是否绑定到正确的 DOM 元素
- **交互流程**：点击、输入、拖拽、快捷键等用户操作的响应
- **前端状态**：操作后 UI 是否实时更新，无需刷新
- **错误展示**：API 返回错误时，页面是否给出合理提示（而不是白屏或 unhandled rejection）
- **边界布局**：极长文本、空列表、加载中状态的显示
- **多 tab / 多窗口**：并发操作时前端状态的一致性

## 2. 功能测试 — UI 交互与前端逻辑

### 2.1 测试一个 PR 的标准流程

1. **读 PR**：看标题、描述、关联 Issue，理解改了什么
2. **看 Changed Files**：
   - 改了 `app/` 或 `components/` → 需要人工验证 UI
   - 改了 `app/api/` → 单元测试已覆盖，重点看对应 UI 是否正确调用
   - 改了 `db/add-*.sql` → 确认新功能在界面上可用，旧数据未受影响
3. **按功能描述操作一遍 Happy Path**
4. **验证状态持久化**：刷新页面，数据还在
5. **跑一遍回归清单**（见 §2.4）

### 2.2 UI 交互检查点

针对**表单和输入**：

- [ ] 空表单提交时，前端校验提示出现在正确的字段旁边（不是 console.error）
- [ ] 提交后按钮进入 loading 状态，不可重复点击
- [ ] 成功后页面跳转或列表刷新，不需要手动刷新
- [ ] 失败后错误提示可读（不是 `[object Object]` 或静默失败）
- [ ] 取消/关闭弹窗后，已填内容是否按预期清空或保留

针对**列表和数据展示**：

- [ ] 空列表时有合适的空状态提示（不是空白区域）
- [ ] 超长文本是否截断或换行，不破坏布局
- [ ] 数据加载中有 loading 指示器
- [ ] 数据加载失败时有错误提示，不是白屏

针对**拖拽 / 排序 / 可编辑表格**（如适用）：

- [ ] 拖拽后顺序在刷新后仍保持
- [ ] 内联编辑聚焦/失焦时保存时机符合预期
- [ ] 快速切换行时不触发意外的并发保存

### 2.3 前端状态正确性

这是人工测试最核心的部分——验证**前端内存状态**与**服务器实际状态**是否一致。

典型检查模式：

```
操作 → 看 UI 变化 → 刷新页面 → 确认持久化结果一致
```

常见的前端状态 bug：
- 乐观更新（optimistic update）成功，但实际 API 失败了，UI 显示成功但数据没变
- 删除一条记录后，列表立即更新，但刷新后它又回来了
- 两个 tab 同时操作同一条数据，一个 tab 显示旧值
- 弹窗关闭后，父组件的数据未刷新

**每次操作后刷新页面一次，是发现这类 bug 最简单的手段。**

### 2.4 回归清单

变更涉及以下模块时，额外验证对应功能未退化：

| 变更区域 | 应额外验证 |
|---------|-----------|
| 鉴权/Session | 登录、登出、刷新后登录状态保持 |
| 剧本编辑器 | 输入、保存、版本切换、评论 |
| 场景/角色列表 | 增删改查、拖拽排序 |
| Cue List | Cue 增删、顺序、关联场景 |
| 资产 | 上传（包括大文件）、预览、挂载 |
| 日程/日历 | 事件创建、发布、通知 |
| 权限管理 | 成员添加/移除、角色修改后立即生效 |

## 3. Adversarial 测试 — 前端对抗性验证

### 3.1 说明

后端层面的越权（401/403）已由单元测试覆盖。本节关注**前端侧**的对抗性问题：

- 前端有没有假设"按钮不可见 = 安全"？
- 网络异常时 UI 是否有合理降级？
- 用户在极端输入下是否能让 UI 崩溃？

### 3.2 前端绕过尝试

**目标**：验证 UI 上隐藏的操作，在后端也有对应的保护，而不是"藏起来就安全"。

操作方式（不需要写代码，用 DevTools 即可）：

1. 打开 Network 面板，找到一个正常操作对应的 API 请求
2. 右键 → "Copy as fetch"
3. 在 Console 里粘贴执行，把参数改为你没有权限操作的资源 ID

```javascript
// 例：把 productionId 换成另一个你不是成员的剧目 ID
fetch("/app/api/production/<other-prod-id>/scenes", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ name: "test" }),
  credentials: "include"
}).then(r => console.log(r.status));

// 预期：403（后端保护，不是前端隐藏）
```

### 3.3 异常输入的 UI 表现

| 操作 | 检查点 |
|------|--------|
| 在文本字段输入 `<script>alert(1)</script>` 并保存 | 保存成功后页面不执行 JS，显示原始文本 |
| 粘贴 10,000 字符的超长文本 | 页面不卡死，有截断或字数限制提示 |
| 上传超大文件（>100MB） | 有明确的大小限制提示，不是浏览器卡死 |
| 上传非预期格式的文件（如把 .exe 改名为 .jpg） | 上传被拒绝或预览降级，不是白屏 |
| 快速连续点击"保存"或"提交"按钮 | 按钮进入禁用状态，只发出一次请求 |

### 3.4 网络异常降级

用 DevTools → Network → Throttling 模拟弱网或断网：

| 场景 | 预期 |
|------|------|
| 请求中断网（Offline） | 出现错误提示，不是白屏；已填内容不丢失 |
| 慢速网络（Slow 3G）下提交表单 | loading 指示器出现，按钮不可重复点击 |
| 服务端返回 500 | 页面给出"请稍后重试"类提示，不是 unhandled rejection |

### 3.5 如何报告发现的问题

在 GitHub Issues 提交，包含：

1. **复现步骤** — 最短能复现的操作序列
2. **实际结果** — 截图 + Network 请求截图（如涉及 API 请求）
3. **预期结果**
4. **严重程度**：
   - 🔴 Critical：绕过权限成功、数据泄露、数据损坏
   - 🟠 High：功能完全不可用、数据丢失
   - 🟡 Medium：功能部分异常、用户体验严重受损
   - 🟢 Low：UI 显示错误、无实际数据危害
